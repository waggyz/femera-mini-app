DRAFT
Femera 0.3 user and developer guide

David Wagner
Abstract
Femera is an open-source finite element-by-element (EBE) matrix-free application that can be tuned for performance across a wide range of problem sizes and HPC architectures. This version (0.3) of the Femera mini-app is for optimizing the basic capabilities of a high-performance finite element analysis toolchain. The primary goal is to verify that the high level of performance demonstrated by version 0.1 can be maintained while providing the flexible data handling emonstrated by version 0.2.

Femera is intended to be built and used in a high-performance parallel computing environment. This guide is organized for different readers.

System administrators: build and install Femera (part I) for use by production and power users.

Production users: run Femera from the command line (part II, section 7) or use it within Python (part II, section 9).

Power users: develop Femera plugins and compile them in a compatible build environment (part III).

Femera developers: maintain, debug, extend, and customize Femera (part IV).

If Femera has already been built for your computing environment, production users may skip ahead to section 7 for details about running Femera from the command line, or section 9 to use Femera in Python.

Part I Build Femera

Femera is not distributed with a ready-to-use binary executable as it is a high-performance computing mini-app that relies on features of the specific hardware it is used on. Femera is only available for Linux and requires a build environment compatible with the intended run-time environment.

7 Start Femera

Femera is an MPI application intended to execute in a hybrid MPI+OpenMP parallel processing environment. The processing environment must be determined before launching the Femera executable with (mpiexec). Installing Femera also installs an executable Bash shell script (from tools/fmrexec.sh) that wraps (mpiexec) for convenience.
fmrexec [RUN_MODE[:fmrmodel]] [-mpi:<MPIOPT>] [-[fmr:]<FMROPT>] […]
This script sets up the parallel environment and starts Femera in one of a variety of different run modes based on the optional (RUN_MODE) keyword. Femera run modes are listed in table 5.

10 Advanced Femera use

Femera is built specifically for each type of CPU. In shared filesystem environments, there should be a different Femera executable file for each different CPU type. The Bash script (tools/fmrmodel.sh) is installed as (fmrmodel) and can be used in an alias to run the version appropriate to the current host.

Best practice: Alias the machine-specific Femera executable.
alias femera="femera-`fmrmodel`"

11 Femera capability life cycle

placeholder
  1. User plugin
    Plugins are detected and loaded at run-time.
  2. External Femera extension
    Extensions may be enabled when configuring Femera before building it. External dependencies may be downloaded and installed automatically by make external. Dependency source code may be available in the repository's (external/) directory or automatically downloaded to (build/). Some external dependencies may be satisfied if they are available on the build system or provided by the user. See section     I for details.
  3. Built-in Femera capability
    Libraries needed must be available on the build system prior to building Femera to enable these capabilities. Examples of these include MPI, OpenMP, and libnuma.
  4. Deprecation
    From time to time a capability may be deprecated. This can happen, for example, if a specific capability can be replaced by a more general capability that performs as well. A capability identified for deprecation will usually still be available as an optional external extension or user plugin until the next major version release.
  5. Removal from repository
    Built-in capabilities identified for deprecation will be removed at the next major Femera version release after beta testing. Though it will not be maintained by Femera developers, it may still be possible to obtain and use the capability as a user plugin.

Part IV Femera theory of operation

Femera is not a replacement for traditional finite element applications. Femera provides capabilities that can not be done efficiently, if at all, by commercial applications. These include the ability to solve very large high-fidelity models, or run a large number of smaller simulations.

  • Very many models

12.2 Test-driven development

placeholder

Best practice: Follow Femera test-driven development steps.
1. Describe a capability in this guide.
2. Add a LaTeX comment starting with (tdd:) and followed by the name of a test file and any arguments needed to verify that the capability has been implemented correctly.
3. Write the test.
4. Check that the test fails with a useful debugging message.
5. Implement and debug the capability until the test passes.

Femera test-driven development (TDD) is started by documenting the desired software behavior in this guide and including a LaTeX comment specifying a file that tests this capability immediately after describing the capability.

Best practice: Test names contain a form of “to be”, “to do”, or “to have”, (e.g. “is”, “are”, “do”, “does”, “did”, “has”, “have”) representing a Boolean (true/false) of the desired behavior. Avoid negative test names (containing, e.g., “does not”, “is not”).

Test naming conventions and syntax are consistent with the best practices for the type of code being tested.
  • Test Bash shell scripts and the Femera executable with Bats [https://bats-core.readthedocs.io/].
  • Test Python scripts with the standard (unittest) module [https://docs.python.org/3/library/unittest.html].
  • Test C/C++ code with GoogleTest [https://google.github.io/googletest/].
